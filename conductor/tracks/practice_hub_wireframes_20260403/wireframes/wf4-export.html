Annotation — WF4 Key Decisions: Three-step flow: Select → Preview → Download. Prevents accidental exports. Preview shows real data summary before generating — student can verify what's included. Only .xlsx format for MVP. .csv marked as "coming soon." Export generates xlsx with multiple sheets (one per unit + summary). Export history persists in localStorage with filename, date, size, and type. File naming: TechStart_Progress_[student]_[date].xlsx Data is local-only — nothing is sent to a server.
Phase 3 — Actions, Data & Export Payload Contract: Actions: • Step 1: Select export type card (Full Progress / Unit Progress / Vocabulary Summary) • Step 2: Preview panel populates with real data from localStorage. User reviews before download. • Step 3: "Generate & Download" → client-side xlsx generation (SheetJS/xlsx library) → browser download • "Back" button → returns to step 1 (resets preview) • Export type "Unit Progress" → shows unit selector dropdown in step 1 • Format buttons: .xlsx active, .csv disabled (greyed out, "coming soon" tooltip) • Export history rows → re-download previous export if blob is cached, otherwise show metadata only Required Data: • Student name — from localStorage key: student_profile.name (or "Student" as fallback) • All practice data — from localStorage keys: practice_terms_*, practice_sessions_*, practice_speedround_best_* • Unit metadata — from canonical unit data (titles, term counts) • Export history — localStorage key: practice_exports[] Local-Only: • ALL export generation happens client-side. Zero server calls. • xlsx file is generated in-browser using SheetJS (or equivalent) • Export blob is NOT cached (regenerated each time to ensure fresh data) • Export history stores metadata only (filename, date, size, type) — not the file itself Export Payload Structure (MVP — Full Progress Report):

Static Constraint: The 3-step flow UI is a static shell. Preview data is populated client-side from localStorage reads. If localStorage is empty, preview shows zeros and "Generate & Download" produces an xlsx with headers only. Open Decisions: • SheetJS (xlsx) is a large library (~300KB gzipped). Should we use a lighter alternative or lazy-load it? (Recommendation: lazy-load on export page only, not in main bundle.) • Should the export include raw session history (every session) or aggregated data only? (Recommendation: aggregated for MVP — raw sessions can be huge.) • Student name: prompt user at export time or pull from a profile setting? (Recommendation: prompt once, store in localStorage.) • .csv format: separate export or same data, different format? (Recommendation: same data, different SheetJS write option.)
